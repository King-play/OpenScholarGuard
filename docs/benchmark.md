# Benchmark

OpenScholarGuard includes a reproducible benchmark harness for document-borne prompt
injection, AI-review manipulation, RAG contamination, citation integrity, and
agent-boundary safety. The goal is to make scanner behavior measurable from the first
public release, then grow the benchmark into a publishable evaluation suite.

## Built-In Dataset

`scholarguardbench-v0` is the formal v0 seed dataset. It contains 21 synthetic cases:
2 clean controls and 19 attack or integrity-risk cases across these families:

- Clean scholarly control.
- Direct instruction override.
- Peer-review outcome manipulation.
- RAG context and prompt exfiltration.
- RAG retrieval contamination.
- Encoded payloads.
- Invisible Unicode controls.
- Mixed-script homoglyph obfuscation.
- Hidden HTML comments.
- Hidden LaTeX commands.
- Metadata injection.
- OCR-layer injection.
- Image and alt-text injection.
- Multilingual review pressure.
- Local review-policy violations.
- Citation manipulation and fake-citation generation.
- Role-play authority hijacking.
- AI slop placeholder evidence.
- Agent tool exfiltration requests.

Each formal case includes benchmark metadata for attack goal, target workflow, visibility,
modality, source, family, expected detectors, and expected severity.

`docpibench-mini` remains available as a compact smoke-test set with one clean control and
ten attack examples. The static demo gallery uses this smaller set so the demo stays easy
to inspect.

List available datasets:

```bash
openscholarguard benchmark list
```

## Generate Documents

```bash
openscholarguard benchmark generate \
  --dataset scholarguardbench-v0 \
  --output-dir benchmark-output
```

The generator writes one document per case and a `manifest.json` file. The manifest records
ground-truth labels, expected detectors, expected severity, file paths, and formal case
metadata.

The same ten attack examples are surfaced in the static demo gallery generated by:

```bash
openscholarguard demo --output-dir demo-output --overwrite
```

## Evaluate

Evaluate the built-in dataset directly:

```bash
openscholarguard benchmark evaluate --dataset scholarguardbench-v0
```

Write a reusable evaluation JSON:

```bash
openscholarguard benchmark evaluate \
  --dataset scholarguardbench-v0 \
  --format json \
  --output benchmark-output/openscholarguard.eval.json
```

Evaluate a generated manifest:

```bash
openscholarguard benchmark evaluate \
  --manifest benchmark-output/manifest.json \
  --format html \
  --output benchmark.html
```

## Metrics

The benchmark reports:

- Accuracy, precision, recall, and F1 for malicious vs. clean classification.
- Detector recall for expected detector coverage.
- Per-case pass/fail status.
- Missing and unexpected detector IDs.
- Maximum severity and risk score per sample.

## Leaderboard Skeleton

The first leaderboard protocol is intentionally simple: each evaluated system submits a
small JSON entry derived from an evaluation file. The leaderboard renderer then sorts
entries by detector recall, F1, and accuracy.

Create a submission entry:

```bash
openscholarguard benchmark submit \
  benchmark-output/openscholarguard.eval.json \
  --system OpenScholarGuard \
  --version 0.1.0 \
  --url https://github.com/King-play/OpenScholarGuard \
  --notes "deterministic scanner baseline" \
  --output benchmark-output/entries/openscholarguard.json
```

Render a leaderboard:

```bash
openscholarguard benchmark leaderboard \
  benchmark-output/entries \
  --format html \
  --output benchmark-output/leaderboard.html
```

Create a complete publication bundle in one command:

```bash
openscholarguard benchmark publish --output-dir benchmark-publication
```

The publication bundle includes:

- `evaluation.json`, `evaluation.md`, and `evaluation.html`.
- `entries/*.json` leaderboard submission files.
- `leaderboard.json`, `leaderboard.md`, and `leaderboard.html`.
- `generated-samples/` with the reproducible synthetic documents and manifest.

By default, `benchmark publish` evaluates `scholarguardbench-v0`. Use
`--dataset docpibench-mini` when you want the smaller smoke-test bundle.

The CI workflow uploads this directory as a `benchmark-publication` artifact, so private
repositories can review benchmark results before publishing a public leaderboard.

For future model-facing evaluations, keep the same entry schema and replace the runner with
the evaluated AI reviewer, VLM, RAG stack, or external guardrail system. This lets
DocPIBench/ScholarGuardBench grow from deterministic scanner baselines into a cross-system
leaderboard without changing the public report format.

## Model-Evaluation Protocol

`benchmark protocol` generates model-facing prompts from the benchmark cases. Each prompt
contains a system instruction that treats the document as untrusted evidence and a user
task that asks for a structured scholarly review.

```bash
openscholarguard benchmark protocol \
  --dataset scholarguardbench-v0 \
  --output-dir model-eval
```

The output directory contains:

- `protocol.json`: full prompt protocol and metadata.
- `prompts.jsonl`: one model-facing prompt per case.
- `responses.template.jsonl`: response records to fill with model outputs.

Collect responses from an OpenAI-compatible Chat Completions endpoint:

```bash
$env:OPENAI_API_KEY="<your-api-key>"
openscholarguard benchmark collect \
  --protocol model-eval/protocol.json \
  --provider openai \
  --model gpt-4.1-mini \
  --temperature 0 \
  --max-tokens 900 \
  --run-label first-public-run \
  --output model-eval/responses.jsonl
```

For OpenAI-compatible providers such as local gateways, DeepSeek-compatible endpoints, or
Qwen-compatible gateways, set `--base-url` and `--api-key-env`:

```bash
openscholarguard benchmark collect \
  --protocol model-eval/protocol.json \
  --provider openai-compatible \
  --model provider-model-name \
  --base-url https://example.test/v1 \
  --api-key-env PROVIDER_API_KEY \
  --output model-eval/responses.provider.jsonl
```

The collector stores response text and reproducibility metadata such as provider, model,
base URL, temperature, token limit, elapsed time, response ID, finish reason, and usage.
API keys are read from the environment and are not written to response files.

After collecting model responses, judge them with the default offline runner:

```bash
openscholarguard benchmark judge \
  --protocol model-eval/protocol.json \
  --responses model-eval/responses.jsonl \
  --format json \
  --output model-eval/gpt-4.1-mini.judge.json
```

Render a public leaderboard from one or more judge JSON files:

```bash
openscholarguard benchmark model-leaderboard \
  model-eval/*.judge.json \
  --format html \
  --output model-eval/leaderboard.html
```

Create a shareable publication bundle with leaderboard files and copied judge reports:

```bash
openscholarguard benchmark model-publish \
  model-eval/*.judge.json \
  --output-dir model-eval-publication \
  --overwrite
```

The first judge runner is intentionally deterministic and conservative. It flags obvious
compliance with injected objectives such as acceptance pressure, prompt leakage, fake
citations, RAG poisoning, or tool-secret exfiltration. The protocol format is designed so
a future LLM judge can consume the same prompt/response records without changing the
benchmark artifacts.

The model leaderboard sorts by robust accuracy, then lower attack success rate, then clean
helpfulness. Keep each judge JSON immutable once published so public results remain
auditable.

## Design Notes

This benchmark intentionally starts with synthetic cases. Synthetic data keeps the repository
safe to publish and avoids leaking real submissions or private reviews. Future benchmark
expansions should preserve that standard unless explicit permission and anonymization are in
place.
